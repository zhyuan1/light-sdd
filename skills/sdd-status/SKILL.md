# sdd-status

Report the current state of an SDD change by scanning its artifact directory. Pure SDD logic -- no delegation to external skills.

---

## Pre-check

1. **Locate change root**: look for `.sdd/changes/` in the project root.
   - If the directory does not exist: report "No SDD changes found. Run `/sdd-propose` to start a new change." and stop.
2. **Resolve target change**:
   - If the user supplied `<change-name>`, verify `.sdd/changes/<change-name>/` exists.
   - If no name supplied and only one change directory exists, use it.
   - If multiple changes exist, list them and ask the user to pick one.

---

## Core Execution

This action has no external delegation. Execute the following scan:

### 0. Active profile

Read `.sdd/config.yaml` in the project root:
- If the file exists and contains `active_profile: <name>`, report that profile name.
- Otherwise report `default`.

Include one line in the status output:
```
Profile: gstack   (or "default" when no profile is set)
```

If the active profile is not `default`, add a one-line summary of the primary framework it uses:
```
Profile: gstack  [routes most actions through gstack skills]
```

### 1. Artifact presence check

Walk the dependency chain and report each artifact:

```
brainstorm.md  -> (optional)
proposal.md    -> (required)
specs/         -> (required, list each <cap>/spec.md)
design.md      -> (optional)
tasks.md       -> (required)
plan.md        -> (optional)
```

For each artifact, report one of:
- present -- file exists and is non-empty
- missing (required) -- required artifact not found
- missing (optional) -- optional artifact not found, acceptable to skip

### 2. Capability cross-reference

If both `proposal.md` and `specs/` exist:
- Parse the Capabilities list from `proposal.md`.
- List each capability and whether a matching `specs/<cap>/spec.md` exists.
- Flag any capability without a spec, or any spec without a matching capability entry.

### 3. Task progress

If `tasks.md` exists:
- Count total tasks (lines matching `- [ ]` or `- [x]`).
- Count completed tasks (lines matching `- [x]`).
- Report completion ratio: e.g. "7 / 12 tasks complete".
- Identify the current batch (first batch with incomplete tasks).

### 4. Review summary

If `reviews/` directory exists:
- List each review file with: type, date, verdict.
- Highlight any `fail` verdicts.

### 5. Phase inference

Based on which artifacts exist and task progress, determine the current phase:

| Condition | Phase |
|---|---|
| No proposal.md | not started |
| proposal.md but no specs/ | proposing |
| specs/ but no tasks.md | specifying |
| tasks.md, 0 tasks done | ready to plan/code |
| tasks.md, some tasks done | coding |
| All tasks done, no verification review | ready to verify |
| Passing verification review | ready to ship |
| Archived | shipped |

---

## Post-check

No artifact is generated by this action. Output:

1. **Status table**: artifact presence, task ratio, current phase.
2. **Recommended next action**: based on phase, suggest the most logical next `/sdd-*` command.

Example output format:

```
Change: user-auth
Profile: gstack  [routes most actions through gstack skills]
Phase:  coding (Batch 2)

Artifacts:
  brainstorm.md    present
  proposal.md      present
  specs/           2/2 capabilities covered
  design.md        missing (optional)
  tasks.md         7/12 tasks complete
  plan.md          present

Reviews:
  spec-review-2026-04-15   pass

Next: run /sdd-code to continue Batch 2, or /sdd-review-code after completing the batch.
```
